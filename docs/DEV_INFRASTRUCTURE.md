## 1. Philosophy

Automate everything that is mechanical and error-prone. Keep everything that requires judgment human. The goal is that a developer who clones the repo on a new machine can be fully productive within five minutes, that no code reaches `master` without passing tests, and that the overhead of doing things right is lower than the overhead of doing things wrong.

This infrastructure is proportional to a small, early-phase project. It is not over-engineered. Every tool in this document earns its place.

## 2. Version Control

**Platform:** GitHub — repo is hosted under a GitHub Team organisation account (required for branch protection enforcement on private repos)
**Branching model:** GitHub Flow — simple, appropriate for a small team

- `master` is always deployable (runnable)
- All work happens on short-lived feature branches
- Branch naming: `feature/<short-description>` or `fix/<short-description>`
  - Examples: `feature/udp-listener`, `fix/multicast-join-error`
- Merge to `master` via Pull Request only — **direct pushes to `master` are blocked by branch protection**
- PRs require all CI checks to pass before merge — **enforced by branch protection**
- Delete branch after merge

**Branch protection (`master`):**

A GitHub ruleset named `master protection` is active on `master`. The following rules are enforced at the GitHub level — not convention, not guidelines:

| Rule | Effect |
|---|---|
| Restrict deletions | `master` cannot be deleted |
| Require a pull request before merging (0 approvals) | No direct pushes; all changes via PR |
| Require status checks to pass (`build` job) | CI must be green before merge |
| Require branches to be up to date | PR branch must include latest `master` |
| Block force pushes | History on `master` cannot be rewritten |

See `TEAM_STRUCTURE.md` section 5 for the full PR workflow and reviewer roles.

**Commit message convention:** imperative tense, present tense, concise
- Good: `Add configurable UDP listener with multicast support`
- Bad: `Added stuff`, `WIP`, `fix`

**.gitignore must include:**
```

*.log
.DS_Store          # macOS
Thumbs.db          # Windows
tools/dumps/*      # repo dump files generated by dump.sh
reviews/           # PR review bundles generated by pr_dump.sh
```

## 3. Python Environment

## 4. Code Quality Automation

### Linter & Formatter: Ruff

A single tool that replaces `flake8`, `isort`, and `black`. Fast, cross-platform, zero config needed beyond a `pyproject.toml` entry.

Install: `pip install ruff` (in `requirements-dev.txt`)

Run manually:
```bash
ruff check src/          # lint
ruff format src/         # format
```

**`pyproject.toml`** configuration:
```toml
[tool.ruff]
line-length = 100
target-version = "py313"

[tool.ruff.lint]
select = ["E", "F", "I"]   # pycodestyle errors, pyflakes, isort
```

Ruff runs automatically in CI on every push. A lint failure blocks merge.

## 5. Testing

**Framework:** `pytest`
**Test location:** `src/tests/` — mirrors the module structure under `src/`
**Methodology:** TDD — tests are written before implementation (red → green → refactor)

### Running tests locally:
```bash
pytest                          # run all tests
pytest src/tests/test_foo.py   # run a specific file
pytest -v                       # verbose output
pytest --cov=src --cov-report=term-missing   # with coverage report
```

### Coverage:
- Tool: `pytest-cov` (in `requirements.txt`)
- Target: 80% minimum line coverage enforced in CI
- Coverage report printed to terminal on every local run
- Coverage failures block merge in CI

### Test categories:
- **Unit tests** — isolated, fully mocked, no network, no filesystem I/O
- **Integration tests** — may touch the filesystem (tmp dirs); never touch the network
- No tests require a live socket or any external service

### Known platform gotcha — Python 3.14 and MagicMock with socket specs

`MagicMock(spec=socket.socket)` constructed **inside** a `patch("socket.socket")` context raises `InvalidSpecError` in Python 3.14. The patch replaces `socket.socket` with a `MagicMock` before the `spec=` argument is evaluated; when Python then tries to introspect the mock-as-spec it fails.

**Fix:** construct the mock socket instance *before* entering the patch context, then pass it via `return_value`:

```python
mock_sock = MagicMock(spec=socket.socket)   # constructed before the patch
with patch("socket.socket", return_value=mock_sock):
    ...
```

## 6. Continuous Integration (CI)

**Platform:** GitHub Actions
**Trigger:** Every push to any branch; every Pull Request targeting `master`

**Workflow file location:** `.github/workflows/ci.yml`

### CI pipeline steps (in order):

1. **Checkout** — clone the repo
2. **Set up Python 3.14.5**
3. **Install dependencies** — `pip install -r requirements.txt -r requirements-dev.txt`
4. **Lint** — `ruff check src/`
5. **Format check** — `ruff format --check src/`
6. **Test + Coverage** — `pytest --cov=src --cov-fail-under=80`

All six steps must pass for the PR to be mergeable. A single failure blocks merge. The required status check name is **`build`** — this is the job name in `ci.yml` and is the check registered in the branch protection ruleset.

**Note:** CI runs on `ubuntu-latest` in GitHub Actions. The application itself runs on Windows and macOS. This is acceptable — Python code is portable and the CI environment validates logic, not OS-specific behaviour.

## 7. Pre-commit Hooks (Local Automation)

CI catches problems before merge. Pre-commit hooks catch them before push — faster feedback, less noise in the PR history.

**Tool:** `pre-commit` (in `requirements-dev.txt`)

**`.pre-commit-config.yaml`:**
```yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.9.10
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
```

Setup (once per machine, after cloning):
```bash
pip install pre-commit
pre-commit install
```

After that, every `git commit` automatically lints and formats the staged files. If Ruff finds an issue it cannot auto-fix, the commit is blocked and the developer sees the error immediately.

## 9. Configuration Management

Application configuration lives in a single `config.yaml` at the project root. It is committed to the repo with safe default values. Sensitive or machine-specific values are never committed.

**`config.yaml` (example — replace with your project's settings):**
```yaml
# [PROJECT NAME] configuration
# Replace these keys with your project's actual configuration values.

  setting_one: "default_value"
  setting_two: 8080

  level: INFO      # DEBUG | INFO | WARNING | ERROR
  to_file: false

```

CLI arguments always override `config.yaml` values. This allows environment-specific configuration without editing the config file.

## 10. Logging

**Library:** Python standard library `logging` — no third-party dependency.

**Format:** `%(asctime)s [%(levelname)s] %(name)s: %(message)s`

**Levels used:**
- `DEBUG` — raw data, internal state
- `INFO` — startup messages, config loaded, application active
- `WARNING` — unexpected input, timeout, recoverable errors
- `ERROR` — I/O failure, config invalid, unrecoverable state

Log level is set in `config.yaml` and overridable via CLI. Default is `INFO`.

**Naming convention:** `ITERATION_N_ANALYSIS.md` where N is the iteration number.

**Created by:** Tech Owner (Claude), based on raw data and observations.

**Committed to:** `docs/` folder, on the feature branch for that iteration, merged to `master` with the rest of the iteration's work.

**Contents (minimum):**

| Section | What it captures |
|---|---|
| Setup | Date, location, hardware/environment active |

| Confirmed unknowns | Which [TBC] items from the TPS are now resolved |

| TPS updates required | Which sections of the TPS need updating |
| Decisions made | Any new technical decisions triggered by findings |

## 13. New Machine Setup (Runbook)

This is the complete sequence to go from a fresh clone to a fully working dev environment. Should take under five minutes.

```bash
# 1. Clone
git clone https://github.com/[ORG]/[REPO].git
cd [REPO]

# 2. Create and activate virtual environment
python -m venv .venv
source .venv/bin/activate        # macOS
.venv\Scripts\activate           # Windows

# 3. Install all dependencies
pip install -r requirements.txt -r requirements-dev.txt

# 4. Install pre-commit hooks
pre-commit install

# 5. Verify everything works
pytest
ruff check src/

# 6. Run the application
python -m src.[module].main
```

**Note for Windows:** Running as a script (`python src/[module]/main.py`) will fail with `ModuleNotFoundError`. Always use the `-m` flag from the project root.

If all steps complete without errors, the machine is ready.

## 15. Developer Tools

### `tools/dump.sh` — Repository Dump

A bash script that traverses the repo and concatenates every git-tracked file into a single text file. For each file it writes the full path and current git commit hash as a header, followed by the file contents.

**Purpose:** Fast Claude catchup. Paste the output into a new Claude chat to restore full project context without relying on memory sync.

**Usage:**

```bash
# Run from project root in Git Bash (Windows) or Terminal (macOS)
bash tools/dump.sh           # dumps entire repo
bash tools/dump.sh src/      # dumps src/ subdirectory only
```

Output file: `<unix-timestamp>.txt` written to `tools/dumps/`. The file is gitignored — it is a throwaway artefact, not a versioned asset.

**What is included:** All files tracked by git (`git ls-files`). Anything in `.gitignore` is excluded.

### `tools/pr_dump.sh` — PR Review Bundle

A bash script that bundles a Pull Request into a single text file for Clead's review. Includes PR metadata, the full diff, existing review comments, and (by default) the full source context from `src/`.

**Usage:**
```bash
bash tools/pr_dump.sh <PR-number>           # includes full src/ context
bash tools/pr_dump.sh <PR-number> --no-src  # diff only (for docs-only PRs)
```

Output: `reviews/pr_<N>_review.txt` — paste into Clead's chat for review.
